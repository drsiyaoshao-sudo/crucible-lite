# Fragmented Amendment Records

Each Amendment lives in its own file: `amendment_NN_slug.md`.
`MANIFEST.md` is the machine-readable index consumed by enforcement checks and the
knowledge graph. `graph.py` reads this file for fast status lookup without loading
every amendment.

## File naming

```
amendment_01_domain_primitives.md
amendment_02_stage_gate_order.md
...
amendment_13_time_domain_validation.md
```

## Required file structure

Every amendment file must start with a standard header block:

```markdown
# Amendment N — Title
*Traces to: Article I | Article II | Article I + II*
*Status: NOT YET RATIFIED | PROPOSED | RATIFIED*

[Full amendment text]
```

Status tokens are exact strings parsed by `graph.py` and `_extract_primitives()`:
- `NOT YET RATIFIED` — Amendment 1 placeholder; no primitives defined yet
- `PROPOSED` — written but not yet ratified by the human engineer
- `RATIFIED` — ratified; remove the `PROPOSED` line and replace with `RATIFIED` status

## Ratifying an amendment

1. Open `amendment_NN_slug.md` and change the Status line from `PROPOSED` to `RATIFIED`
2. Update the Status column in `MANIFEST.md` to `RATIFIED`
3. Commit both files — the pre-commit check will verify the Amendment Index is consistent

## Amendment 1 — special handling

Amendment 1 is generated by `/spec collect`. It is the only amendment written by
a command rather than a human. After `/spec collect` runs:
- `amendment_01_domain_primitives.md` is rewritten with the ratified primitive list
- `MANIFEST.md` row 1 status changes from `NOT YET RATIFIED` to `RATIFIED`
- Both files are staged for the initial project commit

## Relationship to the monolithic amendments.md

`docs/governance/amendments.md` is now a stub index that references this directory.
The canonical source of truth for each amendment is the individual file here.
`graph.py` prefers this directory when it exists, and falls back to `amendments.md`
for backward compatibility with projects that have not yet migrated.
